## 1. Предметная область

Проект реализует поисковый сервис над коллекцией технических статей Wikipedia
(30 статей — языки программирования, БД, ML/AI-темы, DevOps-инструменты).
Пользователь отправляет текстовый запрос, сервис возвращает релевантные
документы с подсветкой совпадений и, опционально, краткую суммаризацию
результатов, сгенерированную LLM (Claude Haiku). Система разделяет
пользователей по ролям: обычный читатель может только искать, писатель может
дополнительно сохранять статьи в «избранное», владелец видит агрегированную
статистику по всем пользователям и управляет ролями.

## 2. Анализ требований

Согласно ТЗ практики (кейс «Поисковик, RAG»), обязательный MVP включает:

| Требование | Реализация |
|---|---|
| Парсинг информационного ресурса | `scripts/parser.py` — Wikipedia API, 30 статей |
| Загрузка данных в поисковые индексы | `scripts/indexer.py` — индекс Elasticsearch `articles` |
| Сервер, принимающий запрос и выдающий документы | `app/main.py` — `GET /search` (multi_match + fuzziness + highlight) |
| Суммаризация результата с помощью LLM | `app/summarizer.py` — Claude Haiku, с экстрактивным fallback без ключа |

Дополнительно (сверх MVP, по общим критериям оценки практики — роли, ER-модель,
SQL-запросы):

- Авторизация (JWT) и три роли: `reader`, `writer`, `owner`
- Реляционная БД (SQLite) для пользователей, аудита поиска и избранного
- Веб-интерфейс (`static/index.html`) с логином, поиском, историей, избранным
  и админ-панелью

## 3. Этапы разработки

1. Парсинг данных из Wikipedia API с бэкоффом на rate-limit (`parser.py`).
2. Индексация в Elasticsearch с полями `title` (boost на уровне запроса),
   `content`, `url`, `source` (`indexer.py`).
3. Базовый поисковый сервер на FastAPI: `GET /search` с подсветкой фрагментов.
4. Суммаризация результатов через Claude с graceful fallback на экстрактивный
   алгоритм (частотный отбор предложений), если ключ API не задан.
5. Простой веб-UI поверх API.
6. Слой пользователей/ролей/аудита: SQLite + SQLAlchemy, bcrypt для паролей,
   JWT для сессий.
7. Разграничение доступа: `/search` требует авторизации, `/saved` — роли
   writer/owner, `/admin/*` — роли owner.
8. UI дополнен экраном логина/регистрации и элементами, зависящими от роли.

## 4. Структура БД

Подробная ER-диаграмма — в `docs/ER-diagram.md`. Кратко: 4 таблицы в SQLite —
`roles`, `users` (FK на `roles`), `search_logs` (FK на `users`, аудит запросов),
`saved_articles` (FK на `users`, избранное). Все связи — 1:N.

Примеры SQL-запросов, использованных в `app/routers/admin.py`
(демонстрируют JOIN, GROUP BY и подзапрос):

```sql
-- JOIN: история поиска всех пользователей с ролью
SELECT u.username, r.name AS role, sl.query, sl.results_count, sl.created_at
FROM search_logs sl
JOIN users u ON u.id = sl.user_id
JOIN roles r ON r.id = u.role_id
ORDER BY sl.created_at DESC;

-- GROUP BY: топ поисковых запросов
SELECT query, COUNT(*) AS count
FROM search_logs
GROUP BY query
ORDER BY count DESC
LIMIT 10;

-- JOIN + GROUP BY + подзапрос: пользователи, ищущие чаще среднего
SELECT u.username, COUNT(sl.id) AS searches
FROM users u
JOIN search_logs sl ON sl.user_id = u.id
GROUP BY u.id, u.username
HAVING COUNT(sl.id) > (
    SELECT AVG(cnt) FROM (
        SELECT COUNT(*) AS cnt FROM search_logs GROUP BY user_id
    )
)
ORDER BY searches DESC;
```

Полнотекстовый индекс Elasticsearch `articles` хранится отдельно от SQLite —
это документное хранилище, не реляционное, поэтому в ER-диаграмму не входит;
его маппинг описан в `scripts/indexer.py`.

## 5. API

Полная спецификация — автосгенерированный Swagger по адресу `/docs` после
запуска сервера. Основные эндпоинты:

| Метод | Путь | Доступ | Назначение |
|---|---|---|---|
| POST | `/auth/register` | публичный | регистрация (роль по умолчанию reader) |
| POST | `/auth/login` | публичный | выдача JWT |
| GET | `/me` | авторизован | текущий пользователь |
| GET | `/search` | авторизован | полнотекстовый поиск + опциональная суммаризация |
| GET | `/history/me` | авторизован | своя история поиска |
| POST/GET/DELETE | `/saved` | writer/owner (создание), любой авторизованный (чтение своего) | избранное |
| GET | `/admin/history` | owner | история поиска всех пользователей |
| GET | `/admin/stats` | owner | агрегированная статистика |
| GET/PATCH | `/admin/users` | owner | список пользователей / смена роли |

## 6. Роли и авторизация

Пароли хэшируются через bcrypt (`app/auth.py::hash_password`), в открытом виде
не хранятся и не передаются повторно. Авторизация — JWT (HS256), токен
передаётся в заголовке `Authorization: Bearer <token>`; при каждом запросе
роль пользователя перечитывается из БД (а не «зашита» в токен), поэтому
изменение роли применяется мгновенно, без переlogина.

Матрица доступа — см. таблицу в `README.md`, раздел «Роли и права».
